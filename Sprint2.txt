Sprint 2 – Booking & Business Rules (Week 3–4)
In Sprint 2, we implemented the booking workflow and the main booking rules. Booking is designed as a two-step approval process: the user submits a booking request for a selected mutual slot, and the request must be approved first by a category administrator, then by the big admin (“admin”). Once the big admin approves a request, the system creates an Appointment, confirms it, and marks the slot as booked.

In addition to the booking workflow, the UI enforces critical constraints such as blocking break-time slots (12:00–13:00), preventing booking past times, and restricting duration/participants input to valid ranges. Business rules were also implemented as strategy classes (e.g., duration, participant limit, overlap prevention) to support rule-based validation and future extensibility.

 User Stories Summary (Booking & Business Rules)
US2.1 – Book appointment (request + approval workflow)
Story: As a user, I want to book an appointment for a specific date and time so that I can reserve a visit/meeting.
Acceptance:

User selects a mutual available slot and submits a booking request.
The selected slot becomes held (not available to others) while the request is pending.
Category admin approves/rejects; if approved → forwarded to big admin.
Big admin approval creates a CONFIRMED appointment and books the slot.
Implemented in:
Service.BookingRequestService.submitRequest(...) (creates request + holds slot)
presentation.UnifiedBookingFrame (UI selection + submit request)
presentation.AdminRequestsFrame (approve/reject UI)
persistence.DataRepository.approveByCategoryAdmin(...) and approveByBigAdmin(...)



US2.2 – Enforce visit duration rule
Story: As a system, I want to limit appointment duration so that no booking exceeds the allowed time.
Acceptance:

Duration must be within allowed limits (not exceeding slot length).
Invalid durations are rejected with an error message.
Implemented in:
presentation.UnifiedBookingFrame (duration input range: 1 .. slotMinutes)
Service.DurationRule (rule class available for validation)



US2.3 – Enforce participant limit
Story: As a system, I want to limit the number of participants per appointment so that capacity rules are respected.
Acceptance:

Participant count must be within the allowed maximum.
If the limit is exceeded, the user is blocked and an error is shown.
Implemented in:
presentation.UnifiedBookingFrame (participants restricted, e.g., 1–5)
Service.ParticipantLimitRule (rule class available for validation)


US2.4 – Prevent booking past time slots
Story: As a system, I want to prevent booking time slots in the past so that users can only request future appointments.
Acceptance:

Past slots are not selectable in the UI.
If a past slot is somehow selected, the system rejects the booking request.
Implemented in:
presentation.UnifiedBookingFrame (renders past slots as “Past - Not bookable”)
presentation.MutualBookingFrame (past slots disabled / blocked)
Service.NotInPastRule (rule class)


US2.5 – Block break interval (12:00–13:00)
Story: As a system, I want to block booking during break time so that users cannot reserve appointments during the defined break interval.
Acceptance:

Slots starting between 12:00 and 13:00 are blocked and not bookable.
The UI shows break time clearly and prevents selection.
Implemented in:
Service.BlockedSlotsRule
presentation.UnifiedBookingFrame + presentation.CompanyAvailableSlotsFrame (break rows highlighted/disabled)


US2.6 – “MAIN + EMERGENCY” booking limit per category (max 2 active)
Story: As a system, I want to limit each user to a maximum of two active bookings (MAIN + EMERGENCY) per category so that capacity is controlled and emergency booking is enforced.
Acceptance:

User cannot exceed 2 active bookings (confirmed + pending requests) within the same category.
After submitting MAIN, the user is required to submit EMERGENCY before closing the booking window.
Implemented in:
Service.BookingRequestService.submitRequest(...) (checks confirmed + pending count)
presentation.UnifiedBookingFrame (enforces required emergency flow)
persistence.DataRepository.countConfirmedForUserCategory(...) and countPendingRequestsForUserCategory(...)


US2.7 – Mark ended appointments as COMPLETED (after time passes)
Story: As a user, I want appointments that have already ended to be marked as COMPLETED so that my bookings list reflects which appointments are finished.
Acceptance:

If an appointment is CONFIRMED and its end time is before the current time, it is displayed as COMPLETED.
COMPLETED bookings remain visible as history (not cancelled).
Implemented in:
domain.AppointmentStatus (added COMPLETED)
domain.Appointment.complete()
presentation.MyBookingsFrame (marks past confirmed appointments as completed before displaying)
