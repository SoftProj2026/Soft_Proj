Sprint 5 — Appointment Types & Polymorphism (Weeks 7–8) Summary In Sprint 5 we added advanced support for multiple appointment types, enforced type‑specific business rules, and implemented immediate emergency notifications. The design uses an enum/polymorphic approach so new types and rules can be added without breaking existing UI or business logic.
___________________________________________________
Sprint 5 User Story Summary :
***************************
US5.1 — Support multiple appointment types
Story: As a user, I want to choose from different types of appointments (e.g., urgent, group, individual, review, emergency).
Acceptance criteria:
The appointment type is stored with the booking and used across booking and modification flows.
UI allows selecting the appointment type during booking.
Implemented in (where to look):
domain/AppointmentType.java (enum representing types)
presentation/AppointmentOptionsFrame.java (type selection UI, if present)
presentation/BookingTypeChoiceDialog.java (entrypoint for choosing booking type)
presentation/UnifiedBookingFrame.java (booking UI that reacts to the chosen type)
_______________________________________________________________
US5.2 — Apply different rules for each type
Story: As a system, I want to enforce type‑specific business rules (participants, duration, etc.).
Acceptance criteria:
Type‑specific rules (min/max participants, allowed duration, other constraints) are enforced.
Validation errors are shown when inputs violate rules.
Implemented in (where to look):
Service/AppointmentTypeRules.java (centralized rule definitions and validators)
Service/AppointmentTypeService.java (method setAppointmentType(...) applies rules when type is set)
Service/BookingRequestService (uses rules when submitting requests)
presentation/UnifiedBookingFrame.java (prompts/validation before submitting request)
___________________________________________________________________
US5.3 — Emergency appointment notifications
Story: As a system, I want emergency appointments to send instant email notifications to user and company.
Acceptance criteria:
Creating/marking an appointment as EMERGENCY triggers an immediate notification email with emergency template/content.
Notification is sent to both the user and the company (and a console fallback exists for testing).
Implemented in (where to look):
Service/AppointmentTypeService.java — emergency handling logic and call to email sender
presentation/BookingTypeChoiceDialog.java — "Emergency Booking" flow that builds emergency message and sends immediately (no booking created)
Service/SmtpEmailSender.java — SMTP implementation (send(from, to, subject, body))
presentation/BookingTypeChoiceDialog.java contains ConsoleEmailSender fallback used during testing
_________________________________________________________________
US5.4 — Polymorphic handling of types
Story: As a developer, I want appointment type logic to support future extensibility via polymorphism/enums.
Acceptance criteria:
Appointment types and rules are extensible; adding a new type or rule does not require widespread changes across UI or services.
Implemented in (where to look):
domain/AppointmentType.java (enum; single source of truth)
Service/AppointmentTypeRules.java (rule factory / polymorphic rule objects)
Service.AppointmentTypeService and Service.BookingRequestService (apply rules polymorphically)
UI: presentation components consult AppointmentType when enabling/disabling fields or deciding prompts
__________________________________________________________________
US5.5 — AI Booking (Suggest mutual slots + send booking request)
Story: As a user, I want the system to suggest the best available mutual time slots using AI logic so I can book faster without manually browsing all times.

Acceptance criteria:
-From the booking type dialog, the user can choose "AI Booking".
- The user enters participants (1–5) and duration (1–60).
- The system suggests up to 5 mutual slots in the same category that satisfy:
  - Slot is in the future.
  - Company slot is available (TimeSlot.isAvailable()).
  - Slot is not break-blocked (12:00–13:00) using BlockedSlotsRule.
  - The user has no overlap with confirmed appointments.
- The user selects one suggested slot.
- The system submits a booking request (PENDING_CATEGORY_ADMIN) and holds the slot for approval.
- If no suitable slots exist, the system shows a clear warning and does not crash.

Implemented in (where to look):
- presentation/BookingTypeChoiceDialog.java
  - Button: "AI Booking (Suggest 5 slots + Send Request)"
  - Flow: openAiSuggestDialog()
- Service/AiBookingAssistantService.java
  - suggestTopMutualSlots(user, category, limit)
  - sendRequestForSlot(user, selectedSlot, duration, participants)
- Service/SmartSlotSuggestionService.java
  - suggest(user, category, limit) (ranking + mutual constraints)
- Service/BlockedSlotsRule.java (break blocking rule)
- Service/BookingRequestService.java
  - submitRequest(...) (holds slot + creates BookingRequest workflow)
